================================================================================
        Infinite Noise TRNG — OpenSSL 3.x Provider: Build Security
================================================================================

This document records the binary-hardening posture of the provider and the
runtime security properties it preserves.  For the underlying design
rationale and threat model, see docs/ARCHITECTURE.txt.


1. BINARY HARDENING
-------------------

The build produces infnoise.so with the following mitigations enabled:

  - Full RELRO          (GOT is mapped read-only after relocation)
  - Stack canary        (-fstack-protector-strong)
  - CET / IBT           (Intel Control-flow Enforcement, indirect-branch
                         tracking)
  - NX                  (non-executable data pages)
  - PIE                 (position-independent executable, ASLR-friendly)
  - FORTIFY_SOURCE=3    (compile-time + run-time checks on libc calls)
  - Stripped            (no symbols leaked in the installed .so)

These are enforced by the Makefile; any regression will surface in the
`make check-hardening` target (see Makefile) or an equivalent
`checksec --file=infnoise.so` run.


2. RUNTIME SECURITY PROPERTIES
------------------------------

  - All entropy buffers are cleansed after use via OPENSSL_cleanse.
  - On error paths, any partial output already written to the caller's
    buffer is cleansed — no stale entropy is ever returned.
  - Dispatch is thread-safe via OpenSSL's CRYPTO_RWLOCK; the provider
    serializes its own context against concurrent callers.
  - DoS prevention: requests are capped at 1 MiB, and the zero-read
    retry loop is bounded at 100 iterations before surfacing an error.
  - Seed buffers live in mlock'd pages allocated with
    OPENSSL_secure_malloc, keeping them out of swap and core dumps.
  - Zeroization is verified in constant time via CRYPTO_memcmp so the
    check itself cannot leak through a timing side channel.


3. REPRODUCIBILITY
------------------

The build is not currently bit-for-bit reproducible.  Known sources
of non-determinism:

  - Compiler version drift.  The Makefile picks `gcc` from PATH; an
    Arch toolchain update changes the compiler binary so a build
    today and the same Makefile + same source tomorrow may produce
    different debug-info and different inline-decision output.

  - Build-id.  `strip --strip-unneeded` is invoked on the final .so
    but does not normalise the SHT_NOTE.gnu.build-id section, which
    embeds a content-hash that depends on the unstripped input.

What IS deterministic:

  - Source files have stable layout; no embedded build dates or
    hostnames.
  - Linker flags and section ordering are stable.
  - SOURCE_DATE_EPOCH is honoured by the toolchain when the caller
    sets it; the Makefile does not export one of its own.

Pursuing reproducible-builds.org-grade reproducibility (compiler
pinning, build-id normalisation, deterministic build wrapper) is
deferred until shortly before the first signed release.  The SBOM
generated by `make sbom` (§4 below) is the immediate supply-chain
integrity priority.

To reproduce a release for verification, match: same Arch base
image, same libcrypto / libftdi1 / libinfnoise versions (recorded
in the SBOM), and any SOURCE_DATE_EPOCH the release was built with.
Diff with `diffoscope`; differences expected only in build-id and
debug-info name mangling.


4. SOFTWARE BILL OF MATERIALS (SBOM)
------------------------------------

`make sbom` produces an SPDX-tag-value SBOM at sbom.spdx.txt covering
the runtime dependencies of the shipped infnoise.so: libcrypto,
libftdi1, libusb-1.0, libinfnoise, glibc.  The output includes each
library's name, version (from pkg-config / ldconfig), and known SPDX
license identifier.

Releases attach the generated SBOM alongside the .so and the
detached PGP signature on the GitHub release page (see
docs/Governance.txt §3 for the signing model).
